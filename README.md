## 1. 데이터 처리 규칙 표

CSV를 grep/awk로 먼저 스캔해 구조적 이상을 찾고, 별도로 "5월 거래 누적 vs 5/31 실사" 대사 대조를 수행해 두 단계로 문제를 발견했습니다. 

### 1-A. 구조적 이상 7건 (파일:라인 근거)

| 파일:라인 | 내용 | 문제 | 처리 |
|---|---|---|---|
| `receipts.csv:14` `R9001,2026-06-20,wd-1002 ,25,9200` | 품목코드 소문자+공백 | `WD-1002`와 문자열 불일치 | **보정** — trim+uppercase 정규화 |
| `receipts.csv:277` `R9002,2026-06-21,XX-9999,10,5000` | 품목 마스터에 없는 코드 | 매칭 불가 | **제외** |
| `receipts.csv:288` `R9003,2026-06-22,TP-4001,30,0` | 단가 0원 | 금액 계산 왜곡 | **보정** — 수량 유지, 금액은 마스터 단가(1,200원) 적용 |
| `receipts.csv:61` `R9004,2026-06-23,GR-5001,40,45000` | 단가 45,000원(마스터 450원의 100배) | 단가 오기입 추정 | **보정** — 수량 유지, 금액은 마스터 단가(450원) 적용 |
| `receipts.csv:35,107,167` `R9005,...,CL-6001,6,3600` (3회 반복) | 완전 동일 행 3중복 | 재고 과다 계상 | **제외** — 최초 1건만 유지 |
| `issues.csv:2` `I9101,2026-06-25,WD-1001,-12` | 출고 수량 음수 | 논리적 오류 | **제외** — 원인 불명 |
| `issues.csv:186` `I9102,2026-06-26,CN-2001,0` | 출고 수량 0 | 무의미 거래 | **제외**(계산 영향 없음, 기록만) |

이상 레코드 수는 8건(제외 5 / 보정 3)입니다 — R9005 중복이 2건의 제외 레코드를 만들기 때문입니다.

### 1-B. 대사 대조로 드러난 의미적 불일치 3건

**전제**: 5/1 시작 재고에 대한 정보가 없기 때문에 시작 재고를 "0"으로 가정해 이론치(5월 입고−출고)를 계산하고 5/31 실사와 대조했습니다. 
8개 품목 중 5개가 일치해, 이 데이터가 실제로 기초재고 0으로 합성되었음을 확인했습니다. 이 과정에서 불일치 3건을 발견하였습니다.

| 품목 | 이론치(5월 입고−출고) | 5/31 실사 | 차이 |
|---|---|---|---|
| WD-1002, CN-2001, TP-4001, GR-5001, CL-6001 | — | — | 0 (정합) |
| **CN-2002** | 103 | 107 | **+4** |
| **TM-3001** | 136 | 134 | **−2** |
| **WD-1001** | 630 | 623 | **−7** |

6월 계산은 5/31 실사값을 출발점으로 사용하기 때문에, 의미적 불일치 3건은 6월 재고 계산에는 전파되지 않습니다.

### 전제: 수량 단위

* 모든 수량(입고/출고/실사)은 품목 기본 단위로 기록되어 있습니다. 
* CN-2001의 이론치(5월 입고−출고=56)가 실사(56)와 변환 없이 정확히 일치합니다. — receipts/issues가 snapshot과 다른 단위였다면 우연히 맞을 수 없으므로, 세 파일 모두 같은 단위를 쓴다고 볼 수 있습니다.
* 따라서 box_qty는 계산에 넣지 않고 화면 표시용 EA 환산에만 사용했습니다.

### 1-C. 완전성 검증 — "더 놓친 이상이 없는가"를 별도로 재점검

§1-A의 7건은 grep으로 발견한 것이라, "grep으로 못 찾는 유형이 더 있지 않을까"라는 질문에 답하기 위해 아래 6가지를 추가로 전수 검사했습니다.

| 검사 | 방법 | 결과 |
|---|---|---|
| 수량/단가 소수점 혼입 | `awk '$4 != int($4)'` | 없음 (전부 정수) |
| **R9xxx 패턴 밖의 단가 불일치 (전 건수 대상)** | 전체 입고를 마스터 단가와 1:1 대조 | **R9003·R9004 2건뿐** — 숨은 단가 이상치 없음을 확정 |
| **ID는 다르지만 (품목,일자,수량) 조합이 같은 은닉 중복** | `(date,item,qty)` 그룹핑 후 3건 이상 그룹 탐색 | **기존에 아는 R9005(3중복) 외 신규 발견 없음** |
| items.csv 마스터 자체 결함 (중복코드/음수가격) | 전수 검사 | 없음 |
| 실사(snapshot) 파일 자체 결함 (중복품목/음수수량) | 전수 검사 | 없음 |
| 수량 통계적 이상치 (IQR 기법, 상한 = Q3+3×IQR) | 입고 342, 출고 111 초과 건 탐색 | 입고 4건이 상한 초과 → 아래 설명 |

**통계적 이상치이지만 정상으로 판단한 사례**:
* IQR 기법이 `2026-05-01`(데이터 첫날)의 입고 8건(EA 품목은 각 300개, BOX 품목은 각 40개)을 이상치로 잡았습니다. 그러나 8개 마스터 품목 **전부** 가 같은 날, 품목 유형별로 일관된 수량으로 입고되어 있어 "오류"가 아니라 **초기 입고(재고 시작 물량)** 로 판단해 유지했습니다. 우연한 수치가 아니라 의도된 패턴이라는 근거입니다.

**"모두 찾았음"을 어떻게 판단했는가**:

* 이상탐지의 완전성을 증명할 방법은 없지만 원리가 서로 다른 두 방법을 병행해 신뢰도를 높였습니다
   
  * ① **구조적 스캔** (§1-A·1-C: grep/awk로 "있으면 안 되는 값"을 컬럼별 제약조건 기준으로 전수 탐지)
  * ② **독립 산술 검증** (§1-B: 5월 거래 누적과 실사를 대조하는 계산 자체가 안 맞으면 역으로 이상을 탐지 — 실제로 구조적 스캔이 못 잡는 CN-2002·TM-3001·WD-1001 3건을 이 방법으로 찾았습니다). 

* 한계: 서로 다른 원리의 두 방법을 상호 보완적으로 사용했으나, 두 방법 모두에 걸리지 않는 유형의 이상은 놓칠 수 있다는 한계는 남습니다.

## 5. 운영 설계 서술 (코드 미구현, 서술)

### ① 매일 새 거래 CSV가 도착하는 경우 — 전체 재계산 없이 반영하는 방안

현재 구조는 브라우저가 매번 전체 CSV를 읽고 처음부터 정제·계산합니다. 운영 환경에서는 이 흐름을 서버의 **일 단위 Node 배치**로 옮기고, 파일 단위 처리 이력(`batch_id`, 파일명, 수신 시각, 처리 상태)과 거래 단위 키(`receipt_id`/`issue_id`, 원본 행 해시)를 저장해 같은 CSV가 다시 들어와도 결과가 중복 반영되지 않도록 합니다.

배치는 신규 행에 기존 정제 규칙(품목코드 정규화, 미등록 품목 제외, 단가 보정, 중복/음수/0수량 판정)을 적용한 뒤 `정상`, `보정`, `제외`, `검토 필요` 상태로 저장합니다. 이후 품목·일자별 입출고 델타를 계산해 `item_code + date` 기준의 일별 재고 집계 테이블에 반영합니다. 과거 일자의 지연 도착 거래가 있으면 해당 일자부터 이후 재고가 달라지므로, 전체 기간을 다시 계산하지 않고 **영향을 받는 시작일 이후의 해당 품목 집계만 재생성**합니다. 현재 `core/` 계산 로직은 순수 함수로 분리되어 있어, 초기 운영 단계에서는 같은 TypeScript 로직을 Node 배치에서 재사용할 수 있습니다.

### ② 데이터가 1일 100만 행 규모로 증가할 경우 — 한계와 대응

현재 설계대로라면 1일 100만 행 규모로 증가할 경우, **브라우저 다운로드 크기, CSV 파싱 시간, JS 배열 메모리, 메인 스레드 점유**에서 병목이 발생합니다. 또한 사용자가 화면을 열 때마다 원본 거래를 O(n)으로 재계산하는 방식은 데이터가 누적될수록 응답 시간이 계속 증가합니다. 따라서 원본 CSV는 클라이언트가 아니라 서버 배치에서 처리하고, 대시보드는 원본 데이터가 아니라 미리 계산된 품목별 현재고·일별 재고·이상 데이터 요약만 조회해야 합니다.

저장 구조는 원본 거래 테이블, 정제 결과 테이블, 품목·일자 단위 재고 집계 테이블로 나눕니다. 집계 테이블은 `item_code + date` 복합키로 인덱싱해 특정 품목의 추이 그래프와 현재고를 빠르게 조회하게 합니다. 초기에는 Node 배치로 현재 TypeScript 계산 규칙을 재사용하고, 데이터 탐색·대량 집계·예측 고도화가 커지는 단계에서는 Python/polars 또는 DuckDB 같은 데이터 처리 도구를 병행하는 방식도 고려할 수 있습니다. 핵심은 원본 대용량 데이터를 브라우저에서 분리하고, 검증 가능한 정제 결과와 조회용 집계를 별도로 유지하는 것입니다.

## 7. AI 활용 기록

**진행 흐름**: 먼저 AI 채팅으로 과제 안내문을 읽고 요구사항을 파악했습니다. 바로 구현하지 않고 `omc-plan` 스킬로 데이터 처리·재고 계산·대사표·예측·화면·운영설계 순서의 구현 계획을 먼저 세웠고, 그 계획을 커밋 단위 체크리스트로 삼아 단계별로 작업했습니다.

| 구분 | 사용 방식                                                                                                                                                                                                                                                                          |
|---|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 사용 도구 | Claude Code(OMC), Claude Design(claude.ai/design), ChatGPT/Codex                                                                                                                                                                                                               |
| 모델 전환 | 작업 성격에 따라 Sonnet(반복적인 구현·테스트 작성)과 Opus(계획 자기평가, 데이터 판단이 필요한 검토)를 전환하며 사용했고, 일부는 ChatGPT를 활용했습니다.                                                                                                                                                                               |
| 권한 설정 | Claude는 편집 수락 모드, ChatGPT/Codex는 승인 요청 모드로 사용했습니다. `항상 허용`은 사용하지 않고, 코드·문서 변경 전 변경 내용을 확인한 뒤 허용했습니다.                                                                                                                                                                           |
| 지시한 작업 | CSV 파싱·정제 규칙 구현, 재고 계산, 합계 대사표, 예측 로직, 테스트 작성, 대시보드 화면 구성, README 문서화 초안을 단계별로 요청했습니다. 각 단계에서는 산출물을 확인하고 이해한 뒤에 커밋하였으며, 필요하면 수정을 요청하였습니다.                                                                                                                                      |
| 디자인 활용 | 대시보드 UI 개선은 **Claude Design**(claude.ai/design)에서 별도로 작업한 목업·핸드오프 패키지를 기준으로, 기존 React 컴포넌트에 재현하는 방식으로 진행했습니다.                                                                                                                                                                  |
| 검증한 사항 | 현재고, 합계 대사표, 5월 대조 불일치, 예측 검증 수치를 `awk`와 자동 테스트로 별도 확인했습니다. 예를 들어 `5/1 기초재고=0` 가정은 다른 방법으로 검증한 뒤 채택했습니다.                                                                                                                                                                       |

## 8. 과제 접근 전략

- 과제에서 화면의 완성도보다 **데이터를 정제하고 검증하는 과정**이 중요하다고 판단했습니다. 그래서 UI는 요구사항 3가지(현재고·추이·이상목록) 범위로 한정해 최대한 단순하게 구성하고, 판단과 검증에 시간을 더 썼습니다.
- 기술 스택은 24시간이라는 시간 제한 때문에 가장 익숙한 TypeScript를 선택했습니다. 데이터 처리에 흔히 쓰이는 Python(pandas 등) 계열 도구에 거부감이 있어서가 아니라, 이번처럼 시간이 제한된 상황에서는 익숙한 스택으로 속도를 확보하는 쪽이 더 합리적이라고 판단했습니다.
- 실행 전략은 "한 번에 완벽하게"가 아니라 **최대한 빨리 골격을 구성한 뒤 검증하면서 수정**하는 방식을 택했습니다. 구현 전에 계획을 세우되, 그 계획도 진행 중간에 스스로 재평가하고 고쳐가며 작업했습니다.
- AI 활용도 같은 원칙을 따랐습니다 — AI로 만들고, 결과를 검증하고, 틀렸거나 과제 범위를 벗어난 부분은 수정하는 **사이클을 반복**했습니다 (자세한 도구·검증 내역은 §7 참고).
