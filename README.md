## 1. 데이터 처리 규칙 표

CSV를 grep/awk로 먼저 스캔해 구조적 이상을 찾고, 별도로 "5월 거래 누적 vs 5/31 실사" 대사 대조를 수행해 두 단계로 문제를 발견했습니다. 

### 1-A. 구조적 이상 7건 (파일:라인 근거)

| 파일:라인 | 내용 | 문제 | 처리 |
|---|---|---|---|
| `receipts.csv:14` `R9001,2026-06-20,wd-1002 ,25,9200` | 품목코드 소문자+공백 | `WD-1002`와 문자열 불일치 | **보정** — trim+uppercase 정규화 |
| `receipts.csv:277` `R9002,2026-06-21,XX-9999,10,5000` | 품목 마스터에 없는 코드 | 매칭 불가 | **제외** |
| `receipts.csv:288` `R9003,2026-06-22,TP-4001,30,0` | 단가 0원 | 금액 계산 왜곡 | **보정** — 수량 유지, 금액은 마스터 단가(1,200원) 적용 |
| `receipts.csv:61` `R9004,2026-06-23,GR-5001,40,45000` | 단가 45,000원(마스터 450원의 100배) | 단가 오기입 추정 | **보정** — 수량 유지, 금액은 마스터 단가(450원) 적용 |
| `receipts.csv:35,107,167` `R9005,...,CL-6001,6,3600` (3회 반복) | 완전 동일 행 3중복 | 재고 과다 계상 | **제외** — 최초 1건만 유지 |
| `issues.csv:2` `I9101,2026-06-25,WD-1001,-12` | 출고 수량 음수 | 논리적 오류 | **제외** — 원인 불명 |
| `issues.csv:186` `I9102,2026-06-26,CN-2001,0` | 출고 수량 0 | 무의미 거래 | **제외**(계산 영향 없음, 기록만) |

이상 레코드 수는 8건(제외 5 / 보정 3)입니다 — R9005 중복이 2건의 제외 레코드를 만들기 때문입니다.

### 1-B. 대사 대조로 드러난 의미적 불일치 3건

**전제**: 5/1 시작 재고에 대한 정보가 없기 때문에 시작 재고를 "0"으로 가정해 이론치(5월 입고−출고)를 계산하고 5/31 실사와 대조했습니다. 
8개 품목 중 5개가 일치해, 이 데이터가 실제로 기초재고 0으로 합성되었음을 확인했습니다. 이 과정에서 불일치 3건을 발견하였습니다.

| 품목 | 이론치(5월 입고−출고) | 5/31 실사 | 차이 |
|---|---|---|---|
| WD-1002, CN-2001, TP-4001, GR-5001, CL-6001 | — | — | 0 (정합) |
| **CN-2002** | 103 | 107 | **+4** |
| **TM-3001** | 136 | 134 | **−2** |
| **WD-1001** | 630 | 623 | **−7** |

6월 계산은 5/31 실사값을 출발점으로 사용하기 때문에, 의미적 불일치 3건은 6월 재고 계산에는 전파되지 않습니다.

### 전제: 수량 단위

* 모든 수량(입고/출고/실사)은 품목 기본 단위로 기록되어 있습니다. 
* CN-2001의 이론치(5월 입고−출고=56)가 실사(56)와 변환 없이 정확히 일치합니다. — receipts/issues가 snapshot과 다른 단위였다면 우연히 맞을 수 없으므로, 세 파일 모두 같은 단위를 쓴다고 볼 수 있습니다.
* 따라서 box_qty는 계산에 넣지 않고 화면 표시용 EA 환산에만 사용했습니다.

### 1-C. 완전성 검증 — "더 놓친 이상이 없는가"를 별도로 재점검

§1-A의 7건은 grep으로 발견한 것이라, "grep으로 못 찾는 유형이 더 있지 않을까"라는 질문에 답하기 위해 아래 6가지를 추가로 전수 검사했습니다.

| 검사 | 방법 | 결과 |
|---|---|---|
| 수량/단가 소수점 혼입 | `awk '$4 != int($4)'` | 없음 (전부 정수) |
| **R9xxx 패턴 밖의 단가 불일치 (전 건수 대상)** | 전체 입고를 마스터 단가와 1:1 대조 | **R9003·R9004 2건뿐** — 숨은 단가 이상치 없음을 확정 |
| **ID는 다르지만 (품목,일자,수량) 조합이 같은 은닉 중복** | `(date,item,qty)` 그룹핑 후 3건 이상 그룹 탐색 | **기존에 아는 R9005(3중복) 외 신규 발견 없음** |
| items.csv 마스터 자체 결함 (중복코드/음수가격) | 전수 검사 | 없음 |
| 실사(snapshot) 파일 자체 결함 (중복품목/음수수량) | 전수 검사 | 없음 |
| 수량 통계적 이상치 (IQR 기법, 상한 = Q3+3×IQR) | 입고 342, 출고 111 초과 건 탐색 | 입고 4건이 상한 초과 → 아래 설명 |

**통계적 이상치이지만 정상으로 판단한 사례**:
* IQR 기법이 `2026-05-01`(데이터 첫날)의 입고 8건(EA 품목은 각 300개, BOX 품목은 각 40개)을 이상치로 잡았습니다. 그러나 8개 마스터 품목 **전부** 가 같은 날, 품목 유형별로 일관된 수량으로 입고되어 있어 "오류"가 아니라 **초기 입고(재고 시작 물량)** 로 판단해 유지했습니다. 우연한 수치가 아니라 의도된 패턴이라는 근거입니다.

**"모두 찾았음"을 어떻게 판단했는가**:

* 이상탐지의 완전성을 증명할 방법은 없지만 원리가 서로 다른 두 방법을 병행해 신뢰도를 높였습니다
   
  * ① **구조적 스캔** (§1-A·1-C: grep/awk로 "있으면 안 되는 값"을 컬럼별 제약조건 기준으로 전수 탐지)
  * ② **독립 산술 검증** (§1-B: 5월 거래 누적과 실사를 대조하는 계산 자체가 안 맞으면 역으로 이상을 탐지 — 실제로 구조적 스캔이 못 잡는 CN-2002·TM-3001·WD-1001 3건을 이 방법으로 찾았습니다). 

* 한계: 서로 다른 원리의 두 방법을 상호 보완적으로 사용했으나, 두 방법 모두에 걸리지 않는 유형의 이상은 놓칠 수 있다는 한계는 남습니다.

## 5. 운영 설계 서술 (코드 미구현, 서술)

### ① 매일 새 거래 CSV가 도착하는 경우 — 전체 재계산 없이 반영하는 방안

현재 구조는 매 로드마다 전체 CSV를 처음부터 재계산합니다. 증분 반영을 위해서는: (1) 품목별 "마지막 처리 일자"를 워터마크로 저장하고, (2) 신규 CSV에서 워터마크 이후 행만 골라 정제 규칙을 적용한 뒤, (3) 그 델타(순증감)만 기존 일별 재고 누적값에 더하는 방식으로 바꿔야 합니다. 재고 계산이 `이전값 + 델타` 형태의 누적 연산이라 구조 변경 없이 증분 방식으로 자연스럽게 옮겨갈 수 있다는 점이 지금 설계의 장점입니다. 정제 규칙(중복/이상치 판정)도 워터마크 이후 신규 데이터에만 적용하면 되므로 재사용 가능합니다.

### ② 데이터가 1일 100만 행 규모로 증가할 경우 — 한계와 대응

현재 설계(브라우저에서 CSV 전량 fetch 후 메모리 파싱)가 가장 먼저 무너지는 지점은 **브라우저 메모리와 메인 스레드**입니다. 100만 행/일이면 수개월 누적 시 CSV 자체가 수백MB~GB급이 되어 파싱이 브라우저 탭을 멈추게 하고, 매 세션 재계산도 불가능해집니다. 대응 방향은 파싱·정제·집계를 서버(배치 또는 스트리밍 ETL)로 이관하고, 정제 결과를 품목·일자 복합키로 인덱싱한 DB에 미리 적재한 뒤, 클라이언트는 필요한 구간만 API로 조회하는 구조로 바꾸는 것입니다. 지금 `core/`의 계산 로직 자체(순수 함수, O(n))는 이 구조로 옮겨도 재사용할 수 있습니다.

## 7. AI 활용 기록

**사용 도구**: Claude Code (Sonnet/Opus)

**작업 방식**: 구현에 앞서 계획을 먼저 세우고, 그 계획을 Claude 스스로 재평가하게 해 허점(기초재고 가정의 근거 부족, 단위 환산 모호성 등)을 먼저 찾아낸 뒤 구현을 시작했습니다. 구현 중에는 매 단계(파싱→정제→재고계산→대사표→예측→화면)를 커밋 단위로 나누고, 커밋 전에 실제로 무엇이 바뀌었는지 직접 확인하는 과정을 거쳤습니다.

**검증한 사항**:
- 데이터 정제 규칙과 재고 계산 결과를 **awk로 원본 CSV를 직접 재집계**해 AI가 계산한 수치(현재고, 합계 대사표, 5월 대조 불일치)와 전부 교차검증했습니다. 특히 "5/1 기초재고=0" 가정은 AI가 처음 제안했을 때 근거가 불충분했으나, 실제 데이터를 대조해보니 8개 중 5개 품목이 정확히 0으로 일치함을 직접 확인하고 나서야 채택했습니다.
- 과제 안내문 원문과 대조해, 화면에 표시할 항목(현재고·추이·이상목록 3가지)과 README에만 넣을 항목(대사표·예측·5월대조)의 경계를 AI의 초기 해석이 아니라 원문 구조를 직접 재확인해 결정했습니다.
- 배포된 사이트를 직접 접속해 실제 렌더링을 확인했습니다.

**수정/조정한 사항**: 초기 계획에서는 대사표와 예측 결과를 대시보드 화면에도 표시하도록 되어 있었으나, 과제 요구사항 3번(대시보드 화면)이 요구하는 3가지 항목에 해당하지 않음을 확인하고 화면에서 제거해 README 산출물로만 남겼습니다.
